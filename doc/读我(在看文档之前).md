# 文档内容说明

## 内部文件夹📁说明

1. api 表示的就是接口，通常是书写UI的人员给书写后端的人员预留的一些接口，UI人员书写出来了那些比较简陋的UI之后就可以查看这个文档进行对应的逻辑书写。
2. architecture 表示的是架构文档，通常内部会拥有一些架构图，虽然比较不为成熟。
3. data.flow.diagram 表示的是数据流图，可能比较简略
4. ui 书写的是UI设计的思路和想法，有时候会存储一些具象化的图片在内部
5. requirement 表示的是需求文档，可以说是最小功能并且最简单的一个文档，人人都能看懂，旨在标记每一个阶段能实现的最小功能（<span style="color:red; font-weight:bold;">重要</span>）  

## 文档的一些特殊标记

(deprecated)表示的是文档已经过时，
可能是因为时间原因废弃，也可能是因为性能太差  
然后没有在原函数上修改而是直接整改了一个库出来  总之原因有很多  
(building...)表示是施工中  
这个和其他的那些文档还有些不一样  
这个文档中的任何内容都不能参考和采纳  
甚至在同文件夹中的东西也不要采纳  

## AI使用说明🤖

文档内部没有使用任何AI工具  
至于为什么那么多的符号，以及换行  
这个不是AI生成的，而是自己去网上搜索的  
并且文档编写人员有Md经验和比较喜欢使用  
这种表情包😃  
因为Linux本身编写的时候，比如书写一个机器人  
其实在github中书写md文件的时候你书写一个: 后面书写一些英文就能摸到对应的图表  
比如就是🅰️ 是 :a 可以查找出来  
之后你就能看到这个机器人的这个图标  
比较方便，打字也能增加一些乐趣  

<mark>**没有使用AI的证据：**  </mark>  
能知道github中的md和其他md渲染方式有一些不一样  
比如说换行就很奇怪  
至少要在每一行之后留下两个空格才行，  
直接换行反而不会成功  
那个看上去很像是AI的文档，就是api（已经过时了，哎😧 当时写了好久的）  
那个是之前手搓对其了半天才弄好的文档
